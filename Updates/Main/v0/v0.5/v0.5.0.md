## Impact
- 建立統一的 capability 描述與驗證機制，使 system plugin 能以標準化方式宣告與提供能力
- 提升 plugin 與能力規範的一致性，透過 capability contract 測試確保實作行為符合宣告的 input/output schema
- 改善 PluginsManager 與 capability registry 的整合，使能力註冊流程在 plugin discovery 階段即可完成
- 提高系統狀態一致性與可靠性，透過原子化 capability 註冊避免 partial state 殘留
- 強化多 PluginsManager 實例隔離，避免共享 capability registry 導致的跨實例狀態污染
- 提升文件完整度與開發者可理解性，提供 capability 使用方式與 schema 規範說明

## Risks & Rollback
- system plugin 若宣告的 capability 與預設能力 ID 不一致或 schema 衝突，plugin 在 discovery 階段將被標記為 invalid 而無法啟用
- 既有 system plugin 若 send() 行為未符合 capability input/output schema，可能在 capability contract 測試中失敗並需調整實作
- capability 註冊流程改為嚴格驗證與原子提交，可能暴露先前未被檢出的 manifest 或 schema 問題
- PluginsManager 預設使用獨立 CapabilitiesManager 實例後，若外部系統依賴共享 capability registry，需改為顯式注入同一 manager
- 如需回滾，可移除 capabilities manifest 欄位並停用 CapabilitiesManager 串接，恢復 PluginsManager 原有 discovery 行為

## Notes
- capabilities 模組目前僅負責能力描述的註冊、驗證與查詢，不涉及能力執行或路由邏輯
- 能力唯一鍵僅以 id 判定，version 僅作描述用途，未實作多版本能力索引
- 僅 system plugin 允許註冊 capability，其他類型 plugin 只能使用既有能力
- capability contract 測試依 testCases 驗證 plugin send() 行為，確保能力宣告與實作保持一致
- capabilities 與未來 capability registry 或路由機制的整合將於後續版本實作
