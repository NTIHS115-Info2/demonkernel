## Impact
- 建立統一的 capability provider 查詢入口，使 system plugins 能以 capability id 取得能力實例並自行調用 send()
- 明確分離 CapabilityManager（能力描述）與 Registry（能力提供者映射）的責任
- 使 pluginsManager 在 plugin 生命週期變化時自動維護 capability 映射，避免過期 provider 被使用
- 提升系統模組化與能力發現機制的可維護性與可測試性

## Tests
- 新增 registry 單元測試，驗證 capability provider 註冊、resolve()、tryResolve()、未知 capability 註冊失敗、重複註冊失敗、無效 provider 驗證失敗，以及 has() / list() 行為
- 擴充 pluginsManager 整合測試，驗證 plugin online 後 provider 會註冊進 registry、可透過 registry 取得 provider 並自行呼叫 send()、offline 後映射會被移除、discover 後舊映射會被清理，以及重複 capability 註冊時第二個 plugin 會啟動失敗

## Risks & Rollback
- pluginsManager 與 capabilityRegistry 的生命週期耦合增加，若 provider 註冊或清理流程錯誤，可能導致 capability 無法被解析或殘留過期映射
- 單一 provider 策略下，若兩個 plugin 宣告同一 capability，第二個 plugin 會啟動失敗，可能影響 plugin 啟動流程

## Notes
- Registry 僅負責 capability provider 的註冊與查詢，不參與能力調度、路由或 send() 代理
- 能力描述與合法性仍由 CapabilitiesManager 管理，Registry 只維護 capability → provider instance 映射
- provider 統一透過 send(input) -> output 作為能力調用入口
- 內部清理 API 僅供 pluginsManager 在 plugin 生命週期管理時使用，不作為對外公開介面
